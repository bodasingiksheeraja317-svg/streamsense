Purpose and Context Incremental — not a reset You have already delivered the foundation. This document does not restart your work — it extends it. Every phase below builds directly on an artifact you have already produced, hardens it to field grade, and connects it to the wider engine. Nothing you have built is discarded. Project STREAMSENSE is OSL's edge signal-classification engine: a real-time pipeline that ingests a continuous stream of samples, extracts features, classifies them, and emits structured results to downstream consumers. The engine is deliberately modality-agnostic — it is validated on open audio, but its interfaces and data structures are designed so that the same compiled code handles any 1-D sample stream without redesign. This Scope of Work covers the remaining sprint (Day 5 → Day 15). It represents approximately two weeks of nominal professional engineering effort, compressed into an intensive ten-working-day window. The compression is achieved by building strictly on your existing, validated artifacts and by freezing interfaces early so that all five tracks can integrate in parallel. 1.1 The Five-Track Engine Five tracks build one engine. Your track is one stage in a pipeline; the others depend on the contract you expose. Treat every interface you publish as a frozen API that four other engineers will compile against. System Interfaces and the Stream Contract All five tracks are decoupled by two frozen contracts and one golden reference set. You must conform to these exactly; conformance is what lets the tracks integrate without a single shared codebase. 2.1 The Generalised Sample-Stream Contract The engine's ingress is a generalised 1-D sample stream. Because the contract is channel-generalised and admits a complex element type, a wideband multi-channel complex stream is simply another set of parameters on the same frozen contract — no interface, buffer, or kernel needs redesign. Build and prove everything on open audio; the generality is structural, not a future rewrite. 2.2 MPIC v1.0 — Frozen Model & Preprocessing Interface Contract MPIC v1.0 (OSL-IPL-2026-INT-002) is FROZEN and governs the Track A ↔ Track B boundary. The validation instantiation is fixed: 16 kHz mono float32 input; 1-second frames; 512-point FFT; 160- sample hop; 64 mel bins; Hann periodic window; centre=False giving 97 time frames; power spectrogram; 10·log10 scaling with 1e-10 epsilon; −80 dB clip floor; output mel \[64, 97\]; global normalisation (mean −30.785545 dB, std 22.157099 dB); model I/O input \[1,1,64,97\] float32 → logits \[1,10\]. Cross-implementation tolerance: max abs error 5e-4 on the normalised tensor, with exact top-1 agreement. Any change requires a new version with joint A+B sign-off. This SOW does not modify MPIC v1.0. It adds capability around the frozen core — additional model heads, a richer egress record, and hardened runtime internals — all backward-compatible with the frozen contract and its golden vectors. 2.3 Golden Vectors (GV1K) — The Cross-Implementation Oracle The 1000-vector golden set (GV1K) is the single source of truth for cross-implementation parity. Track A's reference pipeline and Track B's runtime both validate against it. The current status is 1000 / 1000 PASS for both FP32 and INT8 graphs. Every change you make in this SOW must keep GV1K green; a red GV1K is a hard stop. 3\. Scope of Work — Track A You own the host-side intelligence: preprocessing, the model, feature/embedding generation, and every artifact that Track B compiles into the runtime and that Tracks C and E build upon. The scope extends your existing classifier and ONNX export into a streaming, multi-head, deployment-ready model with a frozen feature contract. 5.1 Foundation already delivered • A working 10-class classifier trained on the open audio corpus (26,984-file training split). • FP32 and QDQ-INT8 ONNX exports, both validated against the golden set. • MPIC v1.0 frozen and co-signed; global normalisation statistics frozen. • GV1K — the 1000-vector golden reference — generated and passing. 5.2 What this SOW adds 1\. Streaming front-end: convert the one-shot 1-second pipeline into a continuous sliding-window framer with overlap, so the model runs on an unbounded stream rather than isolated clips. 2\. Embedding head: expose a fixed-length feature/embedding vector (penultimate-layer or projection head) alongside logits, so Track C can build signatures and Track D can carry them in ERR. 3\. Novelty / open-set head: produce a calibrated novelty score (e.g. energy or max-softmax based) so the engine can flag inputs that belong to no known class. 4\. Source-fingerprint support: provide an embedding space and a reference matcher (approximate nearest-neighbour) that identifies recurring sources by signature similarity. 5\. Concurrent-source counting: a lightweight head (or post-processor) that estimates how many distinct sources are simultaneously present in a frame. 6\. Deployment-grade export: a single, fused, statically-shaped, INT8-ready ONNX graph carrying all heads, with GV1K parity preserved. 6\.
